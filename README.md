# TableLLM

TableLLM 是一个让多个网页大模型围绕同一命题持续讨论的本地圆桌工作台。它负责组织席位、周期、共享上下文、浏览器执行和文件事务，同时保留每个模型的原始回复，避免把讨论压缩成只有固定字段的摘要。

本分支只包含圆桌产品，不包含 web_Agent 正常插件产品，也不复制共享 Core 源码。

![圆桌会议工作台](products/roundtable/docs/assets/roundtable-overview.png)

## 产品定位

TableLLM 适合需要多模型独立判断和相互回应的任务，例如方案评审、技术争议、风险分析、资料比较和需要不同观点的研究讨论。

当前稳定自动化席位是 ChatGPT、DeepSeek 和豆包。Gemini、Qwen、Kimi、GLM、Grok 和 Google AI Studio 已进入提供商目录，但仍标记为计划适配，不能视为稳定自动化能力。

TableLLM 不是把同一个提示词同时发送后简单拼接答案。它维护可恢复的讨论计划、每个席位的线程、周期快照、回复关系、容量状态和最终收束。

## 圆桌如何讨论

1. 用户创建圆桌，选择席位并给出任务。
2. 第一周期中，每个席位基于同一份初始上下文独立发言，避免后完成的模型提前跟随同周期答案。
3. 周期结束后，已完成的原文进入公共会话；下一周期的席位读取相同的新快照。
4. 模型可以回应、质疑、修正或补充。明确点名会记录消息级回复关系，但系统不会推断永久阵营。
5. 没有值得公开补充的内容时，席位可以返回内部 `PASS`，本周期显示为旁听，后续仍可重新加入。
6. 达到周期上限、自然收敛或用户终止后，由“东家”基于公共原文完成收束。

“东家”不是额外调用的主持模型，而是现有席位之一。用户可以通过席位布局指定东家；东家不可用时，调度器会从可用席位中选择临时收束者。

## 两种运行模式

| 模式 | 行为 | 适用场景 |
| --- | --- | --- |
| 讨论 | 同一周期席位使用相同快照独立发言，周期结束后统一进入下一轮 | 并行观点、评审、争论和共识形成 |
| 传递 | 按席位顺序传递上下文，最后回到东家总结 | 接力写作、逐步加工和顺序推演 |

默认模式是“讨论”，默认最大周期为 5，可在工作台中调整。

## 工作台能力

- 自由拖动席位，并把一个席位吸附到东家位置。
- 显示发言、等待、旁听、异常、登录和恢复状态。
- 在公共会话中保留模型原文、周期和消息级回复关系。
- 流式生成期间保持用户当前阅读位置，不强制滚动页面。
- 整轮输出结束且用户不在底部时，显示向下箭头，手动滚动到最新消息。
- 在讨论运行期间把用户补充内容排入“下一周期插话”，发送前可以编辑或撤回。
- 为席位设置默认角色，或只对当前任务设置临时角色。
- 跟踪上下文容量，并在接近阈值时执行席位线程交接。
- 保存压缩上下文与最近原文；压缩结果可审阅、修订，不替代原始公共记录。
- 暂停、继续、终止任务，并对不确定发送、登录失效或浏览器异常提供恢复入口。
- 对模型请求的本地文件修改执行权限检查、事务记录和可验证回滚。

## 系统组成

```text
TableLLM 工作台 :3020
  ↓ 会话、席位、周期、事件流、权限和恢复 UI
圆桌调度器
  ↓ 讨论/传递计划、上下文投影、压缩、重试和收束
浏览器运行时
  ├─ 专用 Chrome CDP :9223
  ├─ Playwright MCP :8931
  └─ 可选兼容扩展通道
@web-agents/local-core
  ↓ 工作区路径、权限、事务、原子写入和文件工具
<workspace>/.web-agents
```

默认生命周期使用专用 Chrome profile、CDP 和 Playwright。兼容扩展是可选通道，独立测试，不是默认启动依赖。圆桌不会启动或依赖插件产品的 `3006/3017` 服务。

## 环境要求

- Windows 10 或 Windows 11。
- Node.js 24 或更高版本。
- Chrome，并允许启动项目专用 profile。
- 已登录准备入席的模型网页账号。
- 本地端口 `3020`、`9223` 和 `8931` 未被其他程序占用。

## 安装与启动

在圆桌分支根目录安装依赖：

```powershell
npm ci
```

推荐使用完整 Windows 启动器：

```powershell
.\products\roundtable\start-roundtable.bat
```

启动器会执行环境和端口检查，启动专用 Chrome、Playwright MCP 和圆桌服务，然后打开：

```text
http://127.0.0.1:3020
```

只启动工作台服务时使用：

```powershell
npm run start:roundtable
```

仅启动服务不会替你准备模型网页。需要真实网页自动化时，优先使用完整 BAT 启动器。

## 浏览器与模型连接

默认 CDP 模式使用 `products/roundtable/data/browser-profile` 下的专用浏览器资料，不占用日常 Chrome profile。首次启动后，需要在该浏览器中登录 ChatGPT、DeepSeek、豆包等准备入席的网页。

创建新圆桌时，运行时会检查提供商页面是否已登录、输入框是否可用，并为席位建立可验证的页面绑定。绑定只接受正确提供商域名和真实存在的标签页；查询参数、片段和凭据不会写入状态接口。

新席位线程会优先复用已打开且空闲的提供商标签页，并进入新对话。页面跳转到登录、验证码或其他提供商后，绑定会失效并在工作台提示重新连接。

兼容扩展模式只信任固定的本地圆桌来源，并要求圆桌使用端口 `3020`。不要把正常 web_Agent 插件当作默认圆桌扩展；兼容包位于 `products/roundtable/compat-extension`。

## 工作区、文件与权限

启动后先选择一个已存在的本地工作区。每个工作区独立保存：

```text
<workspace>/.web-agents/
  sessions/                             会话、公共事件和计划
  config/allowed-directories.txt        当前工作区批准的写入目录
  transactions/                         文件事务和恢复记录
  audit/                                操作审计
  backups/                              回滚备份
```

工作区内的文件修改可以按产品策略自动处理；外部修改需要明确路径依据或用户批准。授权与工作区绑定，不会和 web_Agent 插件共享白名单。

模型产生修改工具调用时，圆桌只允许一个明确的写入执行者持有当前任务。事务会记录修改前状态、哈希、备份和执行 ID；失败时自动回滚已修改路径，后续人工编辑与事务哈希冲突时不会被静默覆盖。

单文件删除受权限和审计控制。递归目录删除不属于公开文件工具。

## 本地端口和数据

| 服务 | 默认地址或端口 | 用途 |
| --- | --- | --- |
| TableLLM | `http://127.0.0.1:3020` | 工作台、API 和事件流 |
| Chrome CDP | `9223` | 专用 Chrome 调试连接 |
| Playwright MCP | `8931` | 浏览器自动化服务 |

产品级浏览器 profile、启动器回执和日志默认位于 `products/roundtable/data`。会话业务数据位于所选工作区的 `.web-agents` 中。

可以在本机通过 `products/roundtable/config/data-root.local.txt` 覆盖产品数据目录。`*.local.*`、浏览器 profile、日志、账号状态和真实会话内容都不应提交 Git。

## 常见问题

**双击 BAT 后窗口关闭**

在 PowerShell 中运行 `.\products\roundtable\start-roundtable.bat` 查看完整错误。检查 Node.js、Chrome 路径以及 `3020/9223/8931` 端口。

**端口 `3020` 已被占用**

先访问 `http://127.0.0.1:3020/api/health` 判断是否已有同一圆桌服务。不要直接终止无法确认身份的监听进程。

**席位显示需要登录或无法连接**

在项目专用 Chrome 中打开对应模型网站，完成登录或验证码，然后在工作台重新连接。日常浏览器中的登录状态不会自动共享给专用 profile。

**模型生成时看不到最新文字**

这是当前阅读保护行为：生成期间不会强制滚动。整轮结束后，使用会话区底部的向下箭头跳到最新消息。

**没有出现文件授权窗口**

确认已经选择工作区、任务仍在运行，并检查请求是否属于工作区内路径。工作区内允许的操作可能不需要额外弹窗；外部修改必须有明确路径或进入授权流程。

**兼容扩展无法连接**

确认圆桌运行在固定端口 `3020`，只加载 `products/roundtable/compat-extension`，并检查扩展状态页。自定义圆桌端口不支持兼容扩展模式。
