# 圆桌原文优先讨论输出设计

日期：2026-07-22
状态：已确认方案 A，待用户审核书面设计

## 背景与问题

圆桌当前要求参与模型只输出固定 JSON 对象，并把回复解析为 `summary`、`claims`、`evidence`、`risks`、`disagreements`、`actions`、`missingEvidence` 等字段。前台和后续轮次又优先使用这些结构化字段，因此用户看到的内容以及模型之间传递的内容都可能是程序重组后的版本，而不是网页模型的原始发言。

用户希望圆桌像正常讨论一样输出，不要求特定格式，并允许程序在后台利用结构化结果辅助上下文压缩。

## 目标

- 普通讨论和自动收束允许模型直接输出自然语言或 Markdown，不再强制固定回复格式。
- 模型网页返回的完整原文保存在 `event.content`，并作为会话的权威内容。
- 对话界面直接展示原文；下一轮模型直接读取原文。
- 结构化解析降级为非阻断的派生分析，只服务于压缩、索引和质量辅助。
- 用户看到的发言与下一位模型收到的发言保持一致。

## 非目标

- 不删除上下文压缩、质量分析或结构化回复解析代码。
- 不迁移或改写历史会话数据；旧回复仍按其实际保存的原文展示。
- 不改变本地工具调用的 JSONL/function-call 协议。工具协议与普通讨论回复格式是两条独立链路。
- 不在本次工作中重新设计圆桌布局、参与者调度或轮次策略。

## 方案选择

采用“原文主链 + 后台派生结构化”的方案。

仅修改前台展示会让模型继续生成 JSON，无法得到自然讨论；彻底移除结构化能力又会削弱长会话压缩和质量辅助。因此原文应成为唯一面向用户和模型讨论的主数据，结构化结果保留为可失败、可丢弃、不可覆盖原文的派生数据。

## 设计

### 1. 提示词与模型输出

普通讨论提示词移除固定 JSON Schema、字段列表和 JSON 示例，改为要求模型直接围绕当前阶段自然回答。阶段目标仍可要求判断、理由、风险和下一步，但不规定标题、字段、列表或 Markdown 结构。

自动收束同样输出自然语言；它可以形成清晰结论，但不要求固定栏目。

需要本地工具时，现有工具调用协议继续生效。工具执行后的最终讨论内容仍回归自然回复。

### 2. 权威数据与派生数据

模型返回内容首先原样进入 `event.content`。除现有安全边界所需的长度限制和提示词回显防护外，程序不改写其语义和结构。

结构化解析可以继续对 `event.content` 做尽力提取，结果保存在 `event.metadata.structuredReply`，解析状态保存在 `event.metadata.structureStatus`。这些字段必须满足以下约束：

- 不替换 `event.content`。
- 解析失败不阻止回复提交、展示或进入下一轮。
- 不因结构化字段缺失而把自然回复标记为无效发言。
- 只能被压缩、索引、质量分析等后台能力消费。

### 3. 对话展示

对话渲染器始终以 `event.content` 作为 Markdown 输入，并沿用现有的解析、HTML 清洗和链接加固流程。

前台不再把 `structuredReply` 转换成“核心判断、风险、行动”等可见栏目，也不再为结构化回复显示“查看原始回复”折叠区。结构化状态如需调试，应留在诊断信息或元数据中，不能出现在主要对话正文中。

### 4. 后续轮次上下文

共享上下文中的回复始终由 `event.content` 生成。程序可以继续附加非侵入式的质量提示，但不能使用 `structuredReply` 重写发言。

上下文压缩仍可读取派生结构化字段，生成较早历史的压缩状态；需要保留的近期原文继续直接来自 `event.content`。因此压缩只影响超出近期窗口的历史表达，不改变当前可见回复和模型刚产生的原文。

### 5. 失败与兼容行为

- 自然文本无法解析为结构化字段：正常提交和展示，后台记录低置信或未解析状态。
- Markdown 解析或清洗运行时异常：沿用安全纯文本回退。
- 回复为空：沿用现有空回复或完成检测错误处理，不伪造结构化摘要。
- 历史回复本身是 JSON：按原文展示，不进行历史数据重写。
- 压缩器缺少派生字段：从可用原文或现有压缩回退路径继续工作，不影响讨论主链。

## 数据流

```text
网页模型原始回复
        |
        +--> event.content（权威） --> 对话展示
        |                         --> 下一轮近期上下文
        |                         --> 本地原文持久化
        |
        +--> 尽力结构化解析（派生、可失败）
                                  --> 上下文压缩
                                  --> 索引与质量辅助
```

## 验收标准

1. 新的普通讨论提示词不包含“只输出 JSON”、固定回复 Schema 或固定字段示例。
2. 模型返回普通段落或 Markdown 时，回复可以正常提交并在对话区原样呈现。
3. 即使结构化解析失败，回复也不会因格式原因被阻断、隐藏或降级为错误消息。
4. 对话区不再用结构化栏目替换原文，也不再显示重复的“查看原始回复”。
5. 下一轮提示词中的近期模型发言来自 `event.content`，而不是 `structuredReply` 的重组文本。
6. 上下文压缩仍能在存在派生结构化数据时使用它，并能在缺失时安全回退。
7. 本地工具调用协议和安全渲染能力保持有效。
8. 相关单元测试、圆桌核心测试和浏览器链路测试通过。

## 验证计划

- 更新提示词测试，确认自然回复要求存在且固定 JSON 要求不存在。
- 更新对话渲染器测试，确认有无 `structuredReply` 时都渲染 `event.content`。
- 更新上下文构建测试，确认后续轮次包含原文且不包含结构化重组标签。
- 增加自然文本结构化解析失败但正常提交的调度测试。
- 保留压缩测试，验证派生结构化信息仍可被后台消费并具有原文回退。
- 运行 `npm run test:roundtable`；若涉及完整浏览器完成检测，再运行对应浏览器测试。

## 需求一致性说明

现有 ADworkflo 当前任务聚焦浏览器扩展 MVP，并未覆盖本次圆桌输出优化。本设计应作为独立圆桌任务进入后续实施计划，避免混入扩展任务规格。实现时只修改与圆桌提示词、回复生命周期、上下文投影和对话渲染直接相关的模块。
