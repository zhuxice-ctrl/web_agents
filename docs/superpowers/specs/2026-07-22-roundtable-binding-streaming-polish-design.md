# 圆桌页面绑定与流式输出体验优化设计

日期：2026-07-22
状态：方案 A 已确认，待用户审核书面设计

## 背景

本次反馈包含一个运行故障和四项相互关联的体验优化：

1. 当前圆桌中的豆包连续缺席，但席位仍显示“线程可用”。
2. 模型生成中的流式内容直接撑高公共会话，阅读位置不断变化。
3. 模型原文允许分段，但“一句话一段”的连续短段落使页面过于松散。
4. 模型表达仍有模板化汇报腔，需要增加自然交流的提示控制。
5. 圆桌中央长期显示第一条议题，不能准确表达当前正在执行或最近发出的命令。

本设计延续“原文优先”原则：程序可以改善呈现、状态和提示词，但不能重写模型最终原文。

## 诊断结论

豆包失败不是登录、验证码或输入框选择器故障，而是当前会话线程与实际 CDP 页面绑定错位。

- 当前会话：`20260722-120944-0622a62b`。
- 当前会话期望的豆包线程键：`20260722-120944-0622a62b:doubao:760dc842-8f50-417c-b36a-ae03168ad611`。
- 浏览器中仍存活并被识别为豆包的页面绑定属于前一个会话 `20260722-120933-05364c4c`。
- 当前线程原先对应的页面租约仍记录为 `BOUND_IDLE`，但对应页面已经不在当前 CDP 页面集合中。
- 两次执行均在发送前失败，错误为 `PROVIDER_PAGE_NOT_BOUND`，`sendState` 为 `NOT_SENT`。
- 会话持久化线程状态仍为 `verified`，因此 UI 错误显示“线程可用”。

根因是“持久化线程状态”和“实时浏览器绑定状态”存在双重事实源。快速创建相邻会话或关闭其中一个标签页后，旧会话页面可能仍然存活，而当前会话的页面已经消失。页面状态接口知道实际绑定缺失，但席位 UI 只读取会话内的旧状态。

## 目标

- 当前会话的席位状态与实时浏览器绑定一致。
- 显式重新连接能安全地把现有提供商页面转交给当前线程，不留下同页双绑定。
- 流式内容在独立、可折叠、内部滚动的临时区域中显示。
- 最终回复继续完整、原样展示，不进行段落重写。
- 通过提示词和紧凑样式减少连续短段落与模板化语气。
- 圆桌中央显示当前执行命令；空闲时保留最近一条用户命令。

## 非目标

- 不自动抢占其他会话仍在执行中的页面。
- 不自动关闭用户打开的提供商标签页。
- 不持久化流式折叠状态或滚动位置。
- 不修改模型最终回复的文本、Markdown 或段落结构。
- 不改变轮次调度、公共账本、上下文压缩和本地工具协议。
- 不在本次工作中重新设计完整圆桌布局。

## 方案选择

采用“实时绑定一致性 + 渐进式流式展示 + 提示与样式轻量约束”。

只修改前端无法解决豆包实际不可用；自动重写原文会破坏原文优先；静默把任意豆包页面分配给当前线程又可能干扰其他会话。选定方案只在用户显式重新连接时转交页面，并让显示状态来自实时事实。

## 设计

### 1. 页面绑定一致性

浏览器管理器继续以 `providerId + threadKey` 作为精确绑定键。会话中的 `thread.status` 只记录最近一次配置结果，不能单独证明页面仍可用。

服务端返回会话或浏览器状态时，为每个席位计算一个实时可用性投影：

- 精确 `threadKey` 存在、页面未关闭、URL 仍属于正确提供商且页面租约有效：`liveBinding = verified`。
- 会话记录为 `verified`，但精确页面不存在或已关闭：`liveBinding = missing`。
- 页面跳转到登录或验证页面：沿用 `waiting_login` 或 `waiting_verification`。
- 只存在同提供商、不同线程的页面：当前席位仍为 `missing`，不能显示可用。

前端席位状态优先读取实时投影。`missing` 显示“需要重新连接”，并保留现有席位菜单中的“重新登录/刷新”操作。

### 2. 显式重新连接与旧绑定清理

用户点击当前会话席位的“重新登录/刷新”后，浏览器管理器可以复用一个同提供商页面，但必须先完成转交：

1. 找到将被复用的页面。
2. 删除内存中所有指向该页面的旧绑定键。
3. 将对应旧租约标记为 `ORPHANED`，保留审计证据。
4. 为当前 `threadKey` 绑定或重新绑定租约。
5. 写入新的页面租约标记。
6. 验证登录、URL 和可用输入框后，才把当前席位标记为 `verified`。

如果旧绑定拥有正在运行的执行租约，则不自动转交，返回明确的“页面正在被另一会话使用”错误。这样避免为了恢复豆包而中断其他会话。

### 3. 发送前检查

发送前继续调用精确 `getPage(providerId, { threadKey })`。如果实时绑定缺失：

- 不执行无意义的第二次相同自动重试。
- 回合进入可恢复状态，并向 UI 返回“需要重新连接豆包页面”。
- 不把持久化 `verified` 当作发送依据。

技术失败仍保留在审计与回合诊断中，但用户界面优先显示可行动的中文说明，而不是英文内部错误。

### 4. 流式折叠卡片

每个运行中的回合继续使用现有 `TurnProgressStore` 保存临时流式文本，但渲染为独立 `<details open>` 卡片：

- 摘要行显示模型、轮次、阶段和“正在生成”。
- 正文容器最大高度约 `220px`，使用 `overflow: auto`。
- 无部分文本时显示现有生成占位动画。
- 有部分文本时继续经过 Markdown 解析、HTML 清洗和链接加固。
- 最终 `turn.completed`、`turn.failed` 或 `turn.absent` 到达后，临时卡片由现有终态同步机制移除。

折叠卡片是运行态视图，不写入公共事件、会话文件或浏览器本地存储。

### 5. 流式自动滚动

自动滚动分成两层：

- 外层公共会话沿用现有“用户原本接近底部时才跟随”的规则。
- 内层流式正文在用户接近自身底部时跟随最新文本。

当用户执行以下任一操作时，内层自动跟随暂停：

- 折叠 `<details>`。
- 向上滚动并离开底部阈值。

用户重新展开或手动滚到底部后恢复跟随。更新 DOM 时不能重建 `<details>` 并丢失用户的展开状态；渲染签名更新只替换必要正文，或者在替换前读取并恢复当前 `open` 状态。

### 6. 段落节奏

最终回复仍直接渲染 `event.content`。视觉层只调整排版：

- 连续普通段落间距从当前约 `.55em` 收紧到约 `.35em`。
- 标题、列表、引用、表格和代码块保持现有区分。
- 行高保持可读，不通过负间距或文本拼接强行合并段落。

提示词同时增加内容层约束：相关句子组成完整段落，不要每句话单独分段；只有真正枚举时才使用列表。这样减少新回复中的连续短段落，同时保证历史原文和特殊 Markdown 不被修改。

### 7. 自然交流提示

普通讨论和自动收束共用以下表达要求：

> 模仿人类语气进行自然、正常的交流。直接回应其他参与者和用户，不要使用模板化汇报腔；相关句子组成完整段落，不要每句话单独分段，只有真正枚举时才使用列表。

该要求不规定标题、字段、列表数量或固定结构，并继续服从“自然语言或 Markdown 均可”的原文输出规则。

### 8. 圆桌中央命令文本

中央文本按以下优先级计算：

1. 最新状态为 `running`、`waiting_recovery` 或 `paused` 的计划 `commandText`。
2. 否则取公共事件中最后一条用户 `command` 的 `content`。
3. 两者都不存在时显示“等待第一条指令”。

会话标题继续显示在中央上方。命令文本显示两到三行，超出部分使用省略号；元素 `title` 属性提供完整文本。发送新命令、计划状态变化、恢复和切换会话时都重新计算。

## 数据流

```text
CDP 实际页面 + 页面租约
          |
          +--> 精确 threadKey 实时投影 --> 席位可用/需重连
          |
用户显式重连 --> 清理同页旧绑定 --> 当前线程新绑定

turn.progress --> TurnProgressStore --> 临时 details 卡片 --> 内部自动滚动
turn.completed/failed/absent ------------------------------> 移除临时卡片

当前计划 commandText / 最近 command event --> 圆桌中央命令

模型最终原文 --> event.content --> 安全 Markdown 渲染
                         提示词只改善新回复语气与段落习惯
```

## 错误与恢复

- 精确页面缺失：显示“需要重新连接”，不继续相同自动重试。
- 页面属于其他运行中会话：拒绝转交，提示先结束该会话或使用新标签页。
- 页面需要登录或人机验证：显示现有等待用户状态。
- 流式 Markdown 解析失败：沿用安全纯文本回退。
- SSE 暂时断开：保留最后一份临时文本；最终会话同步负责清理陈旧进度。
- 用户折叠或上滚：视为主动接管，不强制恢复到底部。

## 验收标准

1. 当前线程页面缺失时，豆包席位不再显示“线程可用”。
2. 显式重新连接同提供商页面后，同一页面只有当前线程绑定，旧租约被标记为失效。
3. 其他会话正在使用页面时，不静默抢占。
4. 发送前缺少精确绑定时，回合给出一次可恢复错误，不进行相同的无效重试。
5. 生成中的部分内容显示在默认展开、可折叠、最大高度约 `220px` 的区域内。
6. 用户位于流式正文底部时自动跟随；折叠或向上滚动后暂停；回到底部后恢复。
7. 最终回复到达后临时卡片消失，正式回复完整显示。
8. 连续普通段落视觉间距收紧，但标题、列表、代码块不受破坏。
9. 新提示词包含自然交流和完整段落要求，不重新引入固定输出格式。
10. 圆桌中央运行时显示当前命令，空闲时显示最近命令，并能查看完整文本。
11. 桌面和移动布局无明显溢出、跳动或滚动陷阱。
12. 圆桌核心、浏览器、启动器和新增前端交互测试通过。

## 验证计划

- 浏览器管理器测试：构造两个线程指向同一页面，验证显式重连清理旧绑定和租约。
- 页面租约测试：验证关闭页面、陈旧 `BOUND_IDLE` 和运行中租约保护。
- 服务与 UI 状态测试：验证实时绑定覆盖持久化 `verified`。
- 调度测试：验证 `PROVIDER_PAGE_NOT_BOUND` 转入一次可恢复状态而非重复重试。
- 流式控制器测试：验证默认展开、内部跟随、用户上滚暂停、回到底部恢复和终态移除。
- 提示词测试：验证自然交流与段落约束存在，固定格式要求不存在。
- UI 合同测试：验证中央命令优先级和紧凑段落样式。
- 使用本机 Chrome 加载 `http://127.0.0.1:3020`，检查桌面和移动视口、控制台、截图与一次不发送消息的交互。
- 运行 `npm run test:roundtable` 并重启本地服务做健康检查。

## 需求一致性

现有 ADworkflo 的 `roundtable-raw-first-output` 任务只覆盖原文优先，不包含页面绑定实时投影、流式折叠和中央命令。本设计应建立独立任务，引用原文优先作为前置约束，不修改当前扩展 MVP 的活动任务，也不把这些变更混入插件产品。
