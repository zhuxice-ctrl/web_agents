# 圆桌会话布局、渲染与生成进度设计

状态：已确认  
日期：2026-07-20

## 目标

在保留现有粉白色视觉和圆桌工作流的前提下，提高公共会话区的可读性和运行反馈：

1. 整个右侧详情栏（计划、文件、审计、上下文）可折叠，并让中间工作区立即占用释放的宽度。
2. 模型回复使用接近 ChatGPT 的无气泡文档排版，支持常用 Markdown，同时保留模型、轮次和状态元数据。
3. 每个正在执行的模型都有独立的动态“内容生成中...”占位；能够获取部分文本时原位更新，不能获取时继续显示占位直至最终结果到达。
4. 讨论模式维持“同一轮不同模型并发、下一轮等待本轮全部结束”的屏障语义，并用测试证明真实并发，而不是仅验证最终数据。

## 范围与边界

- 修改范围限定在 `products/roundtable/**` 以及圆桌工作区依赖声明；不修改插件产品。
- 不改变现有粉白色主题、左侧模型席位、中央圆桌和底部输入区的产品结构。
- 不实现逐 token 流式协议，也不依赖各模型网页提供统一流式 API。
- 部分文本属于瞬时 UI 状态，不写入会话账本、回复文件或审计记录；只有最终回复继续作为权威持久化内容。
- 同一提供方的并发上限仍由现有 `providerConcurrency` 控制，默认不提高到大于 1。
- 传话模式继续串行执行；本设计中的轮内并发仅适用于讨论模式。

## 方案选择

### 方案 A：仅最终结果占位

使用已有 `turn.started`、`turn.completed` 和 `turn.failed` 事件显示或移除占位，不抓取部分文本。实现小且稳定，但用户在长回复期间看不到任何正文进展。

### 方案 B：混合进度（采用）

在方案 A 基础上新增 `turn.progress` 瞬时事件。完成检测轮询拿到变化中的助手文本时，按节流规则发送快照；无法可靠抓取部分文本的提供方仍使用动态占位。该方案兼顾反馈速度与网页适配稳定性。

### 方案 C：完整 token 式流式

为每个提供方建立 MutationObserver 或更细粒度的 CDP DOM 流。视觉上最接近原生网页，但强依赖提供方 DOM，维护成本和错误归因风险过高，本阶段不采用。

## 架构

设计分为四个边界清晰的单元：

1. `detailSidebarController` 只负责详情栏折叠状态、持久化和可访问性。
2. `conversationRenderer` 只负责将持久化回复或瞬时部分文本转成安全的文档式 HTML。
3. `turnProgressStore` 只管理当前浏览器页签中的临时执行状态，并把 SSE 事件归并到对应 `turnId`。
4. 自动化完成检测通过可选 `onProgress` 回调向调度器报告变化文本；调度器把它转换成 `turn.progress` SSE 事件，不改变最终持久化流程。

调度器现有 `Promise.allSettled(turns.map(...))` 已经提供轮内并发和轮间屏障，因此不重写讨论调度算法。实现只补充显式并发测试和进度事件接线。

## 详情栏折叠

### 桌面布局

- 在详情栏标签区加入一个图标折叠按钮，使用现有图标按钮语言并提供 `title` 与 `aria-label`。
- 折叠后给 `.workspace-layout` 添加 `is-detail-collapsed`，第三列变为 `0`，详情栏隐藏且不能接收焦点或指针事件。
- 在工作区右上边缘保留一个固定尺寸的恢复按钮。按钮不占用第三列，因此中央 `.roundtable-workspace` 会立即扩展。
- 不改 `.roundtable-workspace` 内部圆桌/会话列比例规则；外层可用宽度增长后，会话区随其自然增宽。
- 折叠状态写入 `localStorage`，键名为 `web-agents-roundtable:detail-sidebar-collapsed`，再次启动页面时恢复。

### 窄屏布局

当前窄屏把详情栏排到主内容下方。窄屏仍允许隐藏或恢复详情栏，但恢复按钮放在工作区工具区内，不创建零宽的第三列，也不遮挡对话或输入栏。切换断点不清除用户保存的偏好。

## 圆桌与会话区域可调分隔

桌面布局在圆桌舞台与公共会话之间增加一条 `8px` 的竖向分隔轨道。分隔轨道是实际网格列，不覆盖正文；视觉线保持细窄，命中区域使用透明扩展宽度，鼠标悬停和拖动时显示粉色强调。

- 指针拖动实时改变公共会话区占主工作区的比例。
- 会话区最小宽度为 `340px`，圆桌舞台最小宽度为 `360px`，拖动不能突破这两个边界。
- 比例写入 `localStorage`，键名为 `web-agents-roundtable:conversation-width-percent`，刷新和下次启动后恢复。
- 分隔轨道使用 `role="separator"`、`aria-orientation="vertical"` 和动态 `aria-valuenow`。
- 聚焦分隔轨道后，左右方向键每次调整 `2%`；`Home` 恢复允许的最窄会话区，`End` 恢复允许的最宽会话区。
- `900px` 及以下继续使用圆桌、会话、输入区上下排列，隐藏分隔轨道，不应用横向拖动比例。

拖动只改变本地展示，不写入圆桌 session、账本或审计记录，也不影响其他浏览器窗口。

## 会话文档渲染

### 视觉结构

- 每条模型回复使用全宽文章行，不使用聊天气泡、浮动卡片或卡片嵌套。
- 文章头部保留模型名称、轮次、阶段、时间和质量状态；正文位于头部下方并使用稳定的内容宽度、行高和段落间距。
- 用户指令、系统状态和模型回复保持可区分，但通过细分隔线、文字层级和元数据样式区分，而不是大面积色块。
- 颜色继续使用现有粉白色变量；代码块和表格可以使用中性灰背景以提高对比度，不引入新主题。
- 每条消息改为两行结构：模型名称、时间、轮次和阶段位于第一行紧凑元信息栏，正文从第二行开始占满消息全部宽度。取消固定宽度的左侧模型列，使模型名称下方空间被正文自然利用。
- 元信息栏允许换行但不占据独立竖向栏；模型名称和正文之间只保留一个紧凑的垂直间距。

### Markdown 能力

最终回复和可用的部分文本支持：

- 标题、段落、加粗、斜体和删除线；
- 有序/无序列表与任务列表；
- 引用、行内代码和围栏代码块；
- 表格、水平线和链接。

采用成熟 Markdown 解析器与 HTML 清理器，而不是正则拼装 HTML。依赖使用 `marked` 与 `dompurify` 的浏览器构建，并由本地圆桌服务提供固定 vendor 路由，页面不访问 CDN。渲染顺序固定为：原始文本 -> Markdown 解析 -> DOMPurify 清理 -> DOM 插入。

清理规则禁止脚本、事件处理属性、内嵌样式、iframe 和危险 URL 协议。外部链接使用新页签，并带 `rel="noopener noreferrer"`。代码内容保持纯文本语义，本阶段不增加语法高亮依赖。

已有结构化回复继续优先显示结构化字段，但先组合成 Markdown 章节后进入同一安全渲染管线。结构无效或缺失时直接渲染原始回复；诊断原文可保留在折叠的 `details` 中，不能让结构解析失败导致正文消失。

## 瞬时生成进度

### 前端状态

`turnProgressStore` 使用 `turnId` 作为主键，记录：

```js
{
  turnId,
  executionId,
  providerId,
  providerLabel,
  round,
  stage,
  status,
  partialText,
  startedAt,
  updatedAt
}
```

状态流转：

- `turn.started`：创建或重置该轮次的模型占位，立即显示“内容生成中...”及动态省略点。
- `turn.progress`：仅当 `executionId` 与当前项一致时更新 `partialText`；有正文后在原位置使用 Markdown 渲染，并保留低干扰的生成中状态。
- `turn.completed`：先触发一次会话刷新；确认持久化回复已出现后删除临时项，避免占位与最终回复同时闪烁或短暂消失。
- `turn.failed` 或缺席结果：刷新持久化状态并清除对应临时项，显示既有失败/缺席结果。
- SSE 重连或页面刷新：从会话计划中状态为 `running` 的 turn 重建占位。瞬时部分文本允许丢失，最终回复不受影响。
- 切换会话：清空前一会话的所有临时项，防止跨会话串线。

会话区只有在更新前已经接近底部（距底部不超过 96px）时自动跟随新内容；用户正在阅读旧消息时不抢滚动位置。对同一 `turnId` 的更新复用 DOM 节点，避免整条消息闪烁。

### 后端事件协议

新增 SSE 事件：

```js
{
  type: "turn.progress",
  sessionId,
  planId,
  runId,
  turnId,
  executionId,
  providerId,
  providerLabel,
  round,
  stage,
  text,
  at
}
```

`text` 是当前检测到的完整快照，而不是增量补丁，使重排和丢事件后的处理保持幂等。前端把事件统一时间字段 `at` 映射为进度项的 `updatedAt`，不使用本地接收时间覆盖它。事件只在规范化文本发生变化时发送，并限制为每个 turn 最多每 400ms 一次；空文本和少于 2 个字符的噪声不发送。`executionId` 用于拒绝自动重试前一执行实例的迟到事件。

Playwright worker 和扩展 worker 的执行请求都接受可选 `onProgress(snapshot)`。各自已有的响应轮询在得到新的、归属明确的助手候选文本时调用它；调用异常不得中断模型执行。调度器传入回调并发出 `turn.progress`。不支持或无法确认助手归属的适配器不调用回调，前端自然退化到占位模式。

## 并发与轮次语义

讨论模式按以下顺序运行：

1. 从轮次开始前的事件列表创建一次不可变上下文快照。
2. 标记本轮全部 turn 为 running，并发送全部 `turn.started`。
3. 通过 `Promise.allSettled` 并发执行本轮不同模型。
4. 全部执行都进入 fulfilled/rejected 后，逐项持久化最终结果。
5. 发送 `round.completed`，下一轮才可创建上下文快照和启动 turn。

同轮回复不能进入同轮其他模型的 prompt；本轮所有最终回复持久化后，下一轮才能看到它们。模型完成速度不同只影响各自进度和最终展示时间，不改变上下文边界。

## 错误处理

- Markdown 解析或清理异常时，回退为 `textContent` 纯文本展示，不能丢失回复。
- `turn.progress` 丢失、乱序或重复时，以 `executionId` 和 `updatedAt` 拒绝旧快照；最终持久化结果始终覆盖瞬时状态。
- SSE 断开时保留当前占位，重连后根据服务端 session 状态校正；终态 turn 不保留生成中提示。
- 部分抓取文本必须经过现有提供方归属校验，不能把用户 prompt、旧回复或另一模型页面内容作为进度发送。
- 一个模型失败不会取消同轮其他模型；现有显式恢复和缺席逻辑保持不变。

## 测试设计

### 单元与契约测试

1. 详情栏控制器：折叠 class、按钮可访问状态、localStorage 恢复和窄屏行为。
2. 工作区分隔控制器：指针拖动边界、百分比持久化、键盘步进、ARIA 数值和窄屏禁用。
3. Markdown 渲染器：标题、列表、引用、代码、表格、链接；脚本、事件属性和危险协议被清理；解析异常回退纯文本。
4. 进度状态：started 创建、progress 原位更新、旧 execution 丢弃、completed/failed 清除、会话切换清空、重连重建。
5. 完成检测器：只在文本变化时报告，节流生效，回调异常不影响最终捕获。
6. SSE 契约：`turn.progress` 字段完整，且不会写入 session ledger。

### 调度回归测试

使用受控 Promise 屏障记录每个 worker 的开始与释放时刻，明确断言：

- 同一轮所有目标模型都已开始，测试才释放任一模型；
- 在本轮最后一个模型结束前，下一轮没有模型开始；
- 下一轮 prompt 同时包含上一轮全部最终回复；
- 同轮 prompt 不包含同轮其他模型回复。

### 浏览器验收

在 `http://127.0.0.1:3020` 用桌面和窄屏视口验证：

- 折叠/恢复右栏后工作区宽度实际变化，按钮不遮挡内容，刷新后状态保持；
- 拖动圆桌/会话分隔线时两侧宽度实时变化，不能越过最小宽度，刷新后比例保持；键盘方向键也能调整。
- 长标题、代码块、宽表格和长链接不溢出对话区；
- 模型元信息位于正文上方，正文占满整行，模型名称下方不再形成固定空白列；
- 多模型同轮开始时同时出现多个占位，部分文本原位增长，最终回复替换且无重复；
- 用户向上滚动阅读时进度更新不强制跳到底部；
- 粉白色主题、右侧对话区和底部输入栏保持现有位置关系。

## 验收标准

1. 用户可折叠整个右侧详情栏，仅保留一个明确的恢复按钮；中央工作区立即填满释放宽度。
2. 折叠偏好跨页面启动保留，桌面和窄屏均可恢复详情栏。
3. 模型正文按安全 Markdown 文档展示，危险 HTML 不执行，解析失败仍能看到纯文本。
4. 每个运行中的模型立即出现独立动态占位；有可靠部分文本时原位更新，没有时保持占位直到终态。
5. 瞬时进度不污染账本，最终持久化回复是唯一权威结果。
6. 自动化测试证明讨论模式同轮真实并发且轮间存在严格屏障。
7. 圆桌聚焦测试、边界检查和相关浏览器测试通过，现有粉白色视觉与会话持久化行为无回归。
8. 桌面分隔线支持指针和键盘调整并持久化；窄屏保持单列堆叠且无水平溢出。
9. 每条会话正文覆盖模型元信息下方的可用宽度，不再保留固定元数据侧栏。
