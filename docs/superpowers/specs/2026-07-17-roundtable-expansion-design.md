# Web Agents 圆桌扩展与自动化设计

状态：已完成逐节产品确认，等待用户审阅书面规范。

日期：2026-07-17

## 1. 产品定位

Web Agents 的主产品不是浏览器插件本身，而是一个本地多模型圆桌协作系统：

> 将用户已经登录的多个网页大模型组织成可指挥、共享上下文、能够调用本地工具完成任务的圆桌。

产品包含五项核心能力：

1. 模型入席：绑定用户已经登录的网页模型。
2. 指令调度：通过结构化 `@` 标签、目标选择、讨论和传递模式决定谁发言。
3. 共享语境：管理议题、用户指令、模型观点、共识、分歧和证据。
4. 本地执行：通过 MCP 读取和修改工作区文件，把讨论结果落地。
5. 用户主控：工作区选择、暂停、终止、外部写入确认、审计和回撤。

## 2. 组件职责

### 2.1 圆桌网页

圆桌网页是产品大脑，负责：

- 圆桌会话和模型席位。
- 结构化指令路由。
- 讨论轮次和传递顺序。
- 公共上下文和模型已读游标。
- 后台任务状态、会话切换和恢复入口。
- 工作区、文件、执行计划和审计视图。

### 2.2 Chrome 插件与网页适配器

Chrome 插件是浏览器连接器，负责：

- 识别模型网页和编辑器。
- 注入文本、点击发送和捕获回复。
- 上报登录、验证码、页面关闭和内容边界状态。
- 为旧插件手动 MCP 模式保留兼容入口。

插件不保存圆桌的权威状态，也不承担最终权限裁决。

### 2.3 本地控制器与 MCP 网关

本地控制器是状态和执行层，负责：

- 工作区验证和数据目录。
- 圆桌事件、运行检查点和任务队列。
- MCP 工具元数据、权限、事务和回撤。
- 浏览器连接、Playwright CDP 和 Playwright MCP 服务。
- 进程身份、端口、日志和恢复。

## 3. 总体架构

```text
圆桌网页
  |-- 会话/指令/轮次/上下文
  |-- 工作区/文件/审计 UI
  |
本地圆桌控制器
  |-- Session Store
  |-- Context Projector
  |-- Run Scheduler
  |-- Transaction Manager
  |
  |-- Playwright CDP ------> 项目专用 Chrome ------> 各模型专用线程
  |-- Playwright MCP ------> 本地诊断与端到端验证
  `-- MCP Gateway ---------> 本地文件系统
```

圆桌运行以项目专用 Chrome 和 Playwright CDP 为主通道。旧插件不再是圆桌调度的运行依赖，只保留手动 MCP 能力。

## 4. 工作区模型

### 4.1 页面入口

用户进入页面后必须先选择工作区。选择动作通过本地链路验证：

- 路径存在并且是目录。
- 当前用户可以读取。
- 内部数据目录可以创建和写入。
- 本地 MCP 和控制器可用。

仅在网页中填写路径字符串不能视为连接成功。

### 4.2 数据布局

```text
<workspace>/
`-- .web-agents/
    |-- workspace.json
    |-- sessions/
    |-- handoffs/
    |-- artifacts/
    |-- audit/
    `-- backups/
```

用户源文件继续保存在工作区原有位置。会话记录、交接包、审计、产物索引和回撤备份放入 `.web-agents`。

切换工作区时，会话列表、文件视图、审计和执行队列随工作区切换，不能混用。

### 4.3 权限边界

| 操作 | 工作区内 | 工作区外 |
|---|---|---|
| 读取、搜索、目录浏览 | 自动 | 自动并审计 |
| 创建、修改、移动 | 自动事务执行 | 弹窗确认一次 |
| 删除、覆盖大量文件 | 自动快照后执行 | 弹窗确认 |
| 不可回撤或外部副作用 | 额外确认 | 额外确认 |

外部写入确认至少展示目标路径、操作类型、涉及文件、预计修改和回撤方式，并提供“拒绝”“仅允许本次”“允许本次任务”。

## 5. 圆桌会话与模型线程

### 5.1 圆桌会话是一级对象

每个圆桌会话保存：

- 会话名称和工作区。
- 模型席位、节点位置和东家位置。
- 每个模型的专用网页线程。
- 公共事件记录和模型上下文游标。
- 运行、轮次、执行计划、文件事务和审计。
- 会话级设置和默认指令路由。

### 5.2 会话命名与切换

- 新圆桌初始名称为“未命名圆桌”。
- 底部输入框的第一条有效用户消息用于命名会话。
- 顶部当前会话名称是会话选择器，点击后可以跳转到其他圆桌。
- 会话名称可以手动修改，后续对话不会自动改名。
- 运行中的圆桌可以在后台继续，切换只改变当前视图。

### 5.3 新圆桌与新线程

- 创建新圆桌时，当前入席模型默认全部使用新的网页聊天线程。
- 登录状态沿用项目专用 Chrome profile，但不能复用其他任务的聊天历史。
- 模型网页线程是圆桌会话和模型席位的一对一绑定。

### 5.4 中途入席

模型中途入席时始终创建新网页线程，并选择上下文策略：

`同步圆桌上下文`：读取会话开始至当前的公共上下文。

`从现在开始`：把加入时事件设为上下文起点，不读取更早记录。

中途模型从下一轮开始正式参与，不插入正在执行的当前轮。以后可以执行“补同步”。已经发送给网页模型的历史无法撤回；需要彻底清空时必须更换新线程。

## 6. 唯一会话输入区

### 6.1 页面原则

删除左侧“当前任务”、任务目标输入框和“新建会话”大按钮。用户任务和后续指导全部从页面底部唯一输入区发出。

顶部会话选择器属于导航，不是第二个任务输入入口。

### 6.2 结构化发言目标

只有经过确认并转换为结构化标签的模型才是发言者。普通文本中的模型名称只作为内容引用，不能自动改变路由。

```text
@豆包，你参考 gpt 的建议说说你的看法
```

执行结果必须是：

```text
发言者：豆包
文本提及：GPT
GPT 不发言
```

### 6.3 自动补全

- 输入 `@` 时弹出当前圆桌模型和“全体”。
- 输入 `ds`、`deepseek`、`豆包`、`gpt` 等别名时显示候选建议。
- 点击、`Tab` 或 `Enter` 才把候选转换为结构化标签。
- `Esc`、继续输入或忽略候选时保留普通文本。
- 空格接受补全可以作为可选设置，但不能默认开启，避免与中文输入法冲突。
- 模型别名允许用户维护。

文本识别只能提出候选，未经确认不得自动改变发言模型或讨论模式。

### 6.4 发送前执行预览

输入区固定展示：

```text
本轮发言：豆包
模式：单独回复
轮数：1
文本提及：GPT
```

常用的本轮目标、模式和轮数放在输入区附近。自动发送和自动捕获成为默认内部能力。上下文策略、并发、交接阈值和权限进入设置抽屉。

## 7. 执行形态

### 7.1 单独回复

一个模型发言一次。未被选择的模型不接收本轮提示，但该回答写入公共记录，并在其他模型下次发言时可见。

### 7.2 并行观点

多个模型读取同一份轮前上下文并各自回答一次。当前轮内彼此不可见；当前轮全部回答在轮末合并，下一轮全部模型可见。

### 7.3 圆桌讨论

默认三轮采用固定阶段：

1. 独立立论：判断、依据、假设和信息缺口。
2. 交叉讨论：回应其他模型的具体观点，进行同意、反驳或修正，并给出增量内容。
3. 收敛方案：整理共识、保留分歧并形成可执行结论。

一轮时等价于并行观点；两轮为立论加交叉讨论；四轮以上在中间继续交叉讨论，最后一轮收敛。

### 7.4 顺序传递

模型按用户排序逐个执行。每一棒同时收到原始任务、公共上下文、最后成功输出和缺席说明，不能只接收上一棒答案。最后一棒回到东家进行汇报。

### 7.5 东家规则

普通轮次不区分东家身份，仍完全按照结构化 `@` 标签、当前目标或“全体”路由。东家只在配置轮次完成后额外执行一次自动收束。

东家不可用时选择健康模型临时收束，并记录替代原因。

## 8. 自动收束与质量旁路

### 8.1 主流程不阻塞

讨论和传递默认全自动。局部乱码、轻度偏题、重复内容和模型身份漂移不能暂停主流程。原始回答始终保存，配置轮次必须正常完成。

### 8.2 非阻塞质量标记

后台可以标记：

- 可能偏题。
- 疑似模型自我宣传。
- 局部乱码。
- 内容重复。
- 回答截断。
- 提示词回显。

标记不删除原文，只改变后续提示中的可信度表达。所有模型输出默认属于待核验候选观点，不能直接视为已确认事实。

下一轮始终重新注入原始用户目标、当前阶段和已确认公共状态。发现候选观点偏移时，其他模型被要求主动纠正，而不是继续沿用。

### 8.3 自动收束

用户设置的轮数仅表示讨论轮。全部讨论轮完成后，系统自动增加一次不计入轮数的可见收束调用。

收束模型必须：

- 回到原始问题。
- 排除明显无关内容。
- 修复局部乱码造成的语义缺口。
- 汇总有效观点。
- 标明未解决分歧和缺少证据。
- 输出最终结果或完整分歧报告。

系统不强迫虚假共识。未形成统一答案时，完整的分歧报告就是合法结束状态。

### 8.4 技术失败

- 空回答、抓取失败和完全截断自动重试一次。
- 重试仍失败时记录错误占位并继续。
- 讨论内容可以继续流动，但低可信回答不能直接触发本地副作用。

## 9. 共享上下文

### 9.1 权威事件记录

本地事件流是唯一真实来源，包含：

- 用户指令。
- 模型回答。
- 轮次指导。
- 共识、分歧和证据。
- 工具调用和结果引用。
- 暂停、重试、回撤和恢复记录。

模型网页自身聊天历史不是圆桌的权威状态。

### 9.2 增量同步

每个模型席位保存最后已读事件。模型下一次发言时接收：

1. 圆桌规则和当前任务。
2. 当前轮次和阶段目标。
3. 已有共识、分歧和待验证证据。
4. 该席位上次同步后新增的公共事件。
5. 必要的近期原文。

共享上下文表示所有模型使用同一个公共事实源，不表示每轮机械重复完整聊天记录。

### 9.3 容量

必须区分：

- 本地公共记录容量。
- 网页输入框单次传输容量。
- 网页模型线程的上下文容量。

网页模型不会公开真实剩余 token，因此线程容量只能根据已发送和已捕获内容估算，并明确标记“估算”。公共事件同步进度是精确值。

### 9.4 长会话压缩

完整记录始终保留在本地。较早内容可以整理为共识、分歧、证据和决策摘要，近期内容保留原文。页面必须区分原文和压缩摘要，并允许查看和修正。

## 10. 交接包

### 10.1 半自动流程

用户在模型节点或席位菜单点击“交接包”，先预览一次，确认后由系统自动完成：

```text
固定公共上下文快照
-> 创建该模型的新网页会话
-> 注入并发送交接包
-> 等待模型确认
-> 补发交接期间新增事件
-> 验证新线程
-> 原子切换席位绑定
```

登录、人机验证、发送失败或确认超时时保留旧线程，不允许半完成切换。

### 10.2 交接内容

- 圆桌名称和目标。
- 当前参与者。
- 用户关键要求。
- 共识、分歧和证据。
- 当前执行状态。
- 最近原始对话。
- 下一轮任务。
- 上下文截止事件编号。

### 10.3 可视化

模型节点外围显示估算容量环。节点详情显示：

```text
上下文容量：约 72%（估算）
公共记录同步：116 / 120
当前网页线程：18 次交互
最后同步：事件 #116
建议：可继续 / 建议交接 / 尽快交接
```

提醒和紧急阈值可配置。

## 11. MCP 执行、事务和回撤

### 11.1 单写入执行权

所有模型可以读取本地证据和提出修改方案，但一个任务只能有一个写入执行模型：

- 单模型写入任务由该模型执行。
- 多模型任务在自动收束后由东家或配置的执行模型落实。
- 用户明确指定“某模型执行”时由指定模型获得执行权。

### 11.2 工具元数据

权限判断使用工具元数据，而不是工具名称正则：

```text
readOnly
mutating
destructive
reversible
openWorld
idempotent
```

### 11.3 事务

一次任务的写入形成事务，至少保存：

- 事务 ID。
- 用户原始指令和执行模型。
- 工具调用顺序和路径。
- 修改前后哈希和文件差异。
- 快照、执行结果和回撤状态。

任一步失败时停止后续写入，并恢复当前事务已经修改的文件。回撤前重新检查文件哈希；发生后续修改时生成恢复副本，不能静默覆盖。

底部输入区支持 `/撤回`、`/撤回 上一个任务` 和 `/撤回 <事务ID>`。

### 11.4 大文件延后风险

`BE-LARGE-IO-01` 作为后端优化事项延后处理。单次大文件和大型项目失败不能直接归因于异步或并发。后续必须分别验证消息体、超时、上下文、序列化、分块、流式传输、分页、背压、断点恢复和大结果落盘引用。

## 12. 并发和锁

### 12.1 模型线程

- 同一个网页线程一次只能执行一个回合。
- 同一提供商的不同专用网页线程可以并发。
- 每个提供商默认软上限为 3 个并发线程，可配置。
- 收到限流、验证码或拥堵信号后动态退避，不能预先把整个提供商强制串行。

### 12.2 文件锁

写入锁默认精确到规范化文件路径：

- 不同文件可以并发写入。
- 同一文件的后续任务等待并重新校验哈希。
- 多文件事务按稳定顺序获取全部路径锁，避免死锁。
- 移动操作锁源路径和目标路径。
- 删除或移动目录时锁目录子树。
- 路径无法预先确定的批量操作才升级为目录或工作区锁。

外部路径进入全局规范路径锁表，防止不同工作区同时修改同一个外部文件。

## 13. 多会话后台运行

多个圆桌可以同时运行。会话切换只改变视图，不停止后台任务。顶部会话选择器显示：

```text
正在运行
排队等待
已暂停
等待登录/验证
已完成
未读结果数量
```

用户可以单独暂停或终止某个圆桌。

## 14. 故障降级和恢复

### 14.1 单模型线程故障

自动重连并重试两次。仍失败时：

- 多模型讨论：该模型本轮缺席，其他模型继续。
- 传递模式：跳过该棒，下一位接收最后成功输出和缺席说明。
- 单模型任务：等待该模型恢复或用户改派。
- 恢复后的模型从下一轮加入并同步缺席期间上下文。
- 东家失败时使用健康模型临时收束。

### 14.2 全局故障

项目 Chrome、圆桌服务或本地 MCP 中断时：

- 未完成步骤写入持久化检查点。
- 受影响任务进入等待恢复。
- 服务恢复后从未完成步骤继续。
- 发送、捕获和工具调用使用唯一执行 ID，防止重复。
- 已完成回答和已提交事务不能重复执行。

### 14.3 登录和验证码

始终由用户手动处理。对应线程暂停，其他独立线程和圆桌继续运行。

## 15. 页面布局

### 15.1 顶部

- 工作区选择器。
- 圆桌会话选择器。
- 当前会话状态。
- 折叠的本地链路状态。

原始端口信息只在状态详情中显示，不占用主导航。

### 15.2 左侧

只保留模型席位、网页线程状态、入席、离席、交接和登录验证提示。删除当前任务区和自动化设置卡片。

### 15.3 中央圆桌

节点显示运行状态、估算容量环、公共记录同步进度、队列、东家效果和交接提醒。中间圆桌继续拥有最大吸引力，模型节点之间有斥力，东家位置是独立固定吸附点。

### 15.4 对话和右侧面板

- 对话记录展示用户指令、模型回复、轮次、收束和执行结果。
- 右侧保留执行计划、文件和审计视图。
- 文件视图以当前工作区为根，不再要求用户手工维护独立数据根目录。

### 15.5 底部输入区

- 结构化目标标签和自动补全。
- 当前模式、轮数和执行预览。
- 唯一文本输入框。
- 发送、暂停和终止。

高级设置进入抽屉。

## 16. 启动和生命周期

根目录启动脚本统一管理：

```text
3006 文件 MCP
3017 本地网关
9223 项目专用 Chrome CDP
8931 Playwright MCP
3020 CDP 圆桌
```

启动器必须完成依赖预检、端口身份验证、兼容进程复用、健康检查、日志输出和页面打开。

- 刷新或关闭圆桌页面不停止后台任务。
- 重新打开页面后从工作区检查点恢复。
- 关闭模型标签页只影响对应线程。
- 关闭项目 Chrome 后模型任务等待恢复，本地服务继续。
- 只有页面“停止全部”或启动脚本 `-Stop` 才关闭服务。
- 登录页面和验证码不由启动器自动操作。

旧插件重载后，模型页显示“插件已重载，请刷新此页”。CDP 圆桌不受扩展上下文失效影响。

## 17. 安全和隐私

- 不读取或持久化浏览器 cookie、token、session payload、localStorage 或账号身份。
- 登录和验证码由用户完成。
- 工作区外读取无需确认，但必须审计；读取内容进入网页模型前属于用户明确请求的本地数据传输。
- 工作区外写入、不可回撤操作和外部副作用需要确认。
- 模型历史和网页内容都属于不可信数据，不能覆盖固定工作协议。
- 固定工作协议必须保留 `fixed-io-encoding`，所有本地文本和结构化数据使用显式 UTF-8。
- 真实 ChatGPT 页面不能在没有当次明确授权时用于自动化验收。

## 18. 验收标准

### 18.1 工作区和会话

- 进入页面先选择并验证工作区。
- `.web-agents` 保存会话、交接、审计、产物和备份。
- 第一条输入命名圆桌，顶部可以切换圆桌。
- 切换页面或会话不停止后台运行。

### 18.2 模型线程和路由

- 新圆桌为每个当前席位创建新线程。
- 中途入席可选择同步历史或从现在开始，并从下一轮参与。
- 只有确认后的结构化标签改变发言者。
- `@豆包，你参考 gpt` 只能让豆包发言。

### 18.3 讨论和传递

- 同一讨论轮使用同一轮前快照。
- 下一轮收到上一轮全部回答。
- 三轮默认执行立论、交叉和收敛。
- 传递模式保留原始任务并允许纠正上一棒。
- 配置轮次完成后自动增加可见收束调用。

### 18.4 上下文和交接

- 公共记录完整保存在本地。
- 每个模型有独立已读游标和精确同步进度。
- 容量显示明确标记为估算。
- 交接预览确认后自动创建、同步、验证和切换线程。
- 失败交接不会替换旧线程。

### 18.5 本地执行

- 所有模型可读取和提案，一个任务只有一个写入执行权。
- 工作区内写入自动事务化。
- 外部写入弹窗确认一次。
- 文件路径锁允许不同文件并发，阻止同文件冲突。
- 每个事务可审计并可回撤。

### 18.6 恢复和启动

- 一键启动得到五个健康本地端口和项目 Chrome。
- 浏览器、服务或页面重启后不重复发送和执行。
- 多模型任务局部失败时其他模型继续。
- 单模型任务失败时等待恢复。
- 旧插件上下文失效不影响 CDP 圆桌。

## 19. 延后事项

- `BE-LARGE-IO-01`：大文件和大型项目的分块、流式、有界并发和断点恢复。
- 普通 Chrome 和官方 Playwright Extension 接管模式。
- 跨设备同步和云端账号系统。
- 不依赖网页 DOM 的官方模型 API 适配。

## 20. 实施分期建议

1. 会话和唯一输入区：工作区、会话选择器、专用线程和结构化路由。
2. 讨论内核：阶段轮次、增量上下文、自动收束和质量旁路。
3. 执行内核：单执行权、事务、路径锁、外部写入确认和回撤。
4. 恢复内核：统一启动器、检查点、幂等动作、并发和故障降级。
5. 交接与容量：进度可视化、交接包和线程轮换。
6. 后端优化：大文件、项目级检索、流式结果和有界并发。
